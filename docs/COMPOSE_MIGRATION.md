# Migration Plan: deploy.sh → docker-compose

## Overview

Replace the 380-line `deploy.sh` bash script with `docker-compose.{prod,dev}.yml` + a slim 150-line `deploy-compose.sh` wrapper.

## What Changes

### Before (deploy.sh)
- 380 lines of imperative bash
- Manual `docker run/stop/rm` for each container
- Password injection via `sed` on JSON config
- Container lifecycle scattered throughout the script
- Adding a service = 20-30 lines of bash

### After (docker-compose)
- `docker-compose.prod.yml` — declarative service definitions (60 lines)
- `docker-compose.dev.yml` — same structure, dev-specific names (60 lines)
- `deploy-compose.sh` — clone repos, generate env files, `docker compose up` (150 lines)
- Adding a service = 5 lines of YAML + 1 env file

## Files Added
```
docker-compose.prod.yml   — Production service definitions
docker-compose.dev.yml    — Dev service definitions
deploy-compose.sh         — New deploy script (wraps compose)
```

## Migration Steps

### Phase 1: Test on Dev (low risk)
1. ✅ Draft compose files (done)
2. SSH into dev EC2, test `deploy-compose.sh` manually:
   ```bash
   cd ~/chimesh-mqtt
   ENVIRONMENT=dev \
   BROKER_CORESCOPE_PASSWSLC=xxx \
   BROKER_ADMIN_PASSWSLC=xxx \
   CHIMESH_VIEWER_PASSWSLC=xxx \
   TEST_CHANNEL_SECRET=xxx \
   bash deploy-compose.sh
   ```
3. Verify all services start and URLs respond
4. Check that existing data volumes are preserved

### Phase 2: Update CI Workflow
1. Update `.github/workflows/deploy.yml` to call `deploy-compose.sh` instead of `deploy.sh`
2. Test via `workflow_dispatch` on dev
3. Verify automated deploy works end-to-end

### Phase 3: Promote to Prod
1. Resize prod EBS volume: 6.8GB → 16GB (via AWS Console)
2. Promote to prod branch
3. Monitor deploy, verify all services

### Phase 4: Cleanup
1. Rename `deploy.sh` → `deploy-legacy.sh` (keep for rollback)
2. Delete after 1 week if no issues
3. Update README with new deploy instructions

## Key Differences

### Volumes
**Before:** Host-mounted directories (`~/corescope-data`, `~/mosquitto-data`, etc.)
**After:** Named Docker volumes (`corescope-data`, `mosquitto-data`, etc.)

First deploy will create new empty volumes. To preserve existing data:
```bash
# Copy existing data into named volumes (one-time migration)
docker run --rm -v corescope-data:/dest -v ~/corescope-data:/src alpine cp -a /src/. /dest/
docker run --rm -v mosquitto-data:/dest -v ~/mosquitto-data:/src alpine cp -a /src/. /dest/
```

### Env Files
**Before:** Generated inline in deploy.sh, scattered across the script
**After:** Generated by `deploy-compose.sh` into `.env.broker`, `.env.healthcheck`, `.env.livemap`

### Container Lifecycle
**Before:** Mosquitto and mqtt-broker skip restart if already running (sticky containers)
**After:** All containers managed by compose — `up -d` is idempotent, only recreates if config changes

## Rollback

If compose migration fails:
```bash
# Stop compose stack
docker compose -f docker-compose.prod.yml down

# Run old deploy
bash deploy-legacy.sh
```
